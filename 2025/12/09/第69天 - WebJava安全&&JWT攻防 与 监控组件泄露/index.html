jwt

jwt介绍

  • ==JSON Web Token(JWT)==。它遵循 JSON 格式，将用户信息加密到 ==token== 里，服务器不保存任何用户信息，只保存密钥信息，通过使用特定加密算法验证 ==token==，通过 ==token== 验证用户身份。基于 ==token== 的身份验证可以替代传统的 ==cookie+session== 身份验证方法。
  • 这使得 ==JWT== 成为高度分布式网站的热门选择，在这些网站中，用户需要与多个后端服务器无缝交互。

jwt三部分 头部.payload.签名
头部 payload 在数据流中的前缀都是eyj（特征）
1.头部

  • 声明jwt的签名算法
  • ==alg==字段，表示采用的算法，如==HS256==，==RS256==
  • 还有 ==kid==等其他字段
    2.载荷（payload）
  • 储存用户信息，承载各种声明以及数据
  • iss：该字段表示 jwt 的签发者。
  • sub：该 jwt 面向的用户。
  • aud：jwt 的接收方。。
  • exp：jwt 的过期时间，通常来说是一个时间戳。
  • iat：jwt 的签发时间，常来说是一个时间戳。
  • jti：此 jwt 的唯一标识。通常用于解决请求中的重放攻击。该字段在大多数地方没有被提及或使用。因为使用此字段就意味着必须要在服务器维护一张 jti 表， 当客户端携带 jwt 访问的时候需要在 jti 表中查找这个唯一标识是否被使用过。
    3.签名
  • 对==头部==，以及==payload==部分进行签名，防止数据被篡改

jwt利用

==从数据包中获取jwt->对jwt进行解析->修改jwt->将修改后的jwt发送==

攻击：

1.空加密：
头部声明加密算法，如果将头部alg字段设置为”none”(即不进行加密)，此时签名部分会被置空，失去验证数据是否修改的功能，此时任何的token都是有效的

2.爆破密钥（使用工具 例如jwttool）
不是空加密时，想要修改jwt进行身份伪造时，需要密钥此时可用空机密或者利用工具将密钥爆破出来，然后修改jwt

爆破密钥只适合对称加密
非对称爆破只能通过其他方式获取公钥和私钥 例如：泄露源码

3.未校验签名
某些服务端未校验签名，可以尝试修改payload或者删除jwt的签名,再次请求 看返回结果

jwt验证过程：
1.原始头部，payload经过加密算法，加密密钥生成签名
2.服务端通过解密算法，解密密钥还原payload
当采用非对称加密算法加密jwt时，知道加密算法，可以通过修改jwt之后用加密算法进行加密来进行身份伪造;若不知道加密密钥，知道解密密钥，可以通过修改jwt的头部,将alg字段修改为对称加密，此时加密密钥 解密密钥相同，服务端收到jwt时，用解密密钥还原，从而达成身份伪造
